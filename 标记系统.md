# 标记系统

## 研究范围

本文件记录对 `interface/JX` 插件里“标记人 / 选择标记目标”系统的本地研究结果，并给出真传里实现 `Ctrl+1` 到 `Ctrl+5` 标记当前目标、`F1` 到 `F5` 选择对应目标的计划。

主要证据来自这些可读文件：

- `interface/JX/JX_0Base/lang/zhcn.lang`：简体语言表，需要用 GB18030 解码。
- `interface/JX/JX_0Base/lang/zhtw.lang`：繁体语言表，UTF-8 可直接读取。
- `interface/JX/JX_Team/JX_FastMarker.ini`：快速标记面板 UI。
- `interface/JX/JX_TargetList/JX_TargetList.ini`：焦点列表里的标记显示。
- `INTERFACE_EXPERIENCE.md`：以前的界面资源审计，确认多种官方 `.UITex` 图集不在本仓库里。

重要限制：JX 和 MY 的主要 `.lua` 文件在本仓库中是打包/编码后的二进制内容，不是可读 Lua 源码。因此可以确认 UI 控件、语言名、引用的图集路径；但不能从本地直接确认运行时 Lua 如何把“云、剑、斧头”等标记映射到具体 frame。

## JX 标记系统结论

JX 的相关功能叫“快速标记&选择列表”。语言表证明它支持这些操作和限制：

- 标记目标：`标记目标为[%s]`
- 选择标记：`选择标记[%s]`
- 一键快速标记：`一键快速标记`
- 清除所有标记：`清除所有标记`
- 权限检查：`你没有标记权限！`
- 数量限制：`标记数量已达上限！`
- 场景限制：`当前地图禁止使用标记功能！`
- 组队限制：`只有组队时才会显示。`

这说明它不是单纯的本地 UI 小标签，而是接在游戏的团队/队伍标记权限体系上。语言表还出现“团队的队长、分配者、标记者方可发起集火/救急指令”，进一步说明标记和团队指令系统有关。

```text

```

因此，“标记人”的核心图标资源是：

```text
ui\Image\button\FrendNPartyButton.UITex
```

它不是仓库里的单张 PNG，也不是 `白云.png` 这种独立图片，而是 JX3 客户端官方 UI 图集中的某个 frame。

## 如果标记某人为“白云”

本地语言表里没有单独的“白云”文本，只有“云 / 雲”。实际玩家口中的“白云”应当对应这套标记里的“云”。

已确认的回答：

- 标记名：`云`。
- 使用资源：`ui\Image\button\FrendNPartyButton.UITex`。
- 显示控件：快速标记面板用 `Image_MarkIcon`，焦点列表用 `Image_FocusMark`。
- 显示尺寸：两个位置都是 24x24 左右。
- 本仓库没有 `FrendNPartyButton.UITex` 的原始图集或 frame 坐标文件。
- 运行时到底把“云”设置成哪个 frame，很可能在打包后的 JX Lua 中完成；该 Lua 在本仓库不可读。

需要谨慎的点：`JX_FastMarker.ini` 里 `Image_MarkIcon` 的默认 frame 是 `77`，`JX_TargetList.ini` 里 `Image_FocusMark` 的默认 frame 是 `66`，MY 的一些焦点/团本 UI 默认也出现过 `67`。这些默认值能证明它们都来自同一个官方图集，但不能单独证明“白云一定是 frame 77/66/67”。准确 frame 需要官方 `FrendNPartyButton.UITex` 的 atlas/frame 表，或可读的运行时 Lua 标记表。

## 真传实现计划

目标：实现 5 个可绑定目标槽位。`Ctrl+1` 到 `Ctrl+5` 把当前选中的目标写入对应标记槽；`F1` 到 `F5` 选择对应标记槽里的目标。

### 1. 数据模型

新增 5 个目标标记槽。建议先做成对全战局可见的 `GameState.targetMarks`，而不是只存在前端本地状态，这样其他玩家也能看到标记图标，后续也能扩展成真正的团队标记。

建议结构：

```ts
type TargetMarkSlot = 1 | 2 | 3 | 4 | 5;

interface TargetMark {
  slot: TargetMarkSlot;
  selection: TargetSelection;
  assignedByUserId: string;
  updatedAt: number;
}
```

`selection` 复用现有 `TargetSelection`：`self`、`player`、`entity`。第一版可以只允许 `player` 和 `entity`，避免把自己标成团队目标。

### 2. 后端接口

新增后端路由，例如：

```text
POST /api/game/target/mark
```

请求体：

```json
{ "gameId": "...", "slot": 1, "selection": { "kind": "player", "userId": "..." } }
```

后端处理要点：

- 复用现有 `normalizeTargetSelectionPayload()` 的校验思路，确保目标存在、还活着、不是不可选中状态。
- `selection: null` 或同槽重复标记同一目标时，可以设计为清除该槽。
- 写入 `state.targetMarks[slot]` 后递增 `state.version`，广播 diff。
- 当玩家死亡、进入不可选中、伪装/隐身导致目标选择被清理时，同步清理指向该玩家的标记槽。现有 `clearTargetSelectionsTargetingPlayer()` 是可参考的入口。

### 3. 前端快捷键

现在 `BattleArena.tsx` 已有热键系统，默认绑定格式支持：

- `C+1` 表示 Ctrl+1。
- `F1` 表示 F1。

需要扩展：

- `HotkeyTabId` 增加一个“目标标记”页。
- `HotkeyActionKind` 增加 `target-mark` 和 `target-select`。
- `HOTKEY_ACTION_IDS` 增加 `target-mark:0..4` 和 `target-select:0..4`。
- `DEFAULT_HOTKEY_BINDINGS` 增加：
  - `target-mark:0` 到 `target-mark:4` = `C+1` 到 `C+5`
  - `target-select:0` 到 `target-select:4` = `F1` 到 `F5`
- `parseHotkeyActionId()` 支持新 action kind。
- `triggerHotkeyBinding()` 中处理：
  - `target-mark:n`：取当前 `selectedTargetRef` 或 `selectedEntityRef`，调用后端标记接口。
  - `target-select:n`：从当前状态的 `targetMarks[n]` 读目标，调用现有 `setSelectedTargetId` / `setSelectedEntityId`，并让现有 `onTargetSelection` 同步到后端。

关键修正：当前 `onDown` 会在很早的位置拦截 `F1` 到 `F10` 并直接 `return`，导致后面的 `F1` 选目标逻辑不可达。实现 `F1` 到 `F5` 时，要把目标标记快捷键处理放在这个浏览器功能键拦截之前，或者改成只 `preventDefault()` 不提前返回给游戏逻辑。

### 4. UI 显示

第一版建议先显示轻量标记：

- 在目标头顶/血条附近显示 5 个槽位之一的小标签。
- 在目标框和目标列表里显示同样标记。
- 槽位可以先映射到 JX 前 5 个名字：`云`、`剑`、`斧头`、`钩子`、`红鼓`。

资源策略：不要直接复制 JX3 官方 `FrendNPartyButton.UITex`，本仓库也没有该图集。第一版可以用文字/数字徽标实现；如果之后要做 1:1 风格，需要用户提供可授权的 atlas 导出，或明确授权制作真传自己的标记图标。

### 5. 交互规则

建议规则：

- `Ctrl+1..5`：当前选中玩家/实体写入槽位。
- 当前没有选中目标时按 `Ctrl+1..5`：提示“当前没有可标记目标”。
- 当前目标已经在该槽时再次按同一快捷键：清除该槽。
- `F1..5`：选择对应槽位目标。
- 槽位为空时按 `F1..5`：提示“该标记为空”。
- 槽位目标死亡、消失、不可选中时：自动清除，或选择时提示“标记目标已失效”并清除。

### 6. 测试计划

| 层级 | 要测什么 |
| --- | --- |
| 后端 | 标记目标、清除目标、非法 slot、非法目标、不可选目标、死亡/隐身清理 |
| 前端 | `normalizeKeyboardHotkey()` 能识别 `C+1..C+5` 和 `F1..F5` |
| 前端 | `F1..F5` 不再被浏览器功能键拦截提前吞掉 |
| 前端 | 标记后目标头顶/HUD 出现槽位标记，选择槽位能切换当前目标 |
| 端到端 | 双人或多人战斗中，一个玩家标记目标，其他客户端能看到一致标记 |

## 未解决项

1. 官方“云/白云”标记的精确 frame 编号尚未确认。
2. 本仓库没有 `ui\Image\button\FrendNPartyButton.UITex` 原始图集，也没有对应 `.txt` frame 表。
3. JX 的运行时 Lua 是打包/编码内容，无法直接读取它调用的官方标记 API 或 frame 映射表。
4. 已尝试公开代码搜索：GitHub code search API 需要认证，另一个公开搜索接口临时 503；没有找到可用的明文 JX 标记表。

## 最短结论

JX 标记某人为“白云”时，本质上是使用“云”标记，从官方图集 `ui\Image\button\FrendNPartyButton.UITex` 中取一个 frame 来显示。这个仓库能确认图集路径和 UI 控件，不能确认“云”对应的精确 frame。真传要实现 `Ctrl+1..5` / `F1..5`，最稳的路径是复用现有 `TargetSelection`，新增 5 个战局标记槽，扩展 BattleArena 热键系统，并先用文字/数字标记显示，等官方 atlas 或授权图标明确后再换成图像资源。
